# Lattice

## What This Is

This is the compass. Every decision — by a human or a minion — runs through this document. If something we're building doesn't align with what's written here, we stop and correct course. This is how we don't drift.

## Who We Are

We build for AI agents the way Apple builds for humans.

Apple asked: "What does a human need to do their best work?" and then obsessed over every detail of that answer — the screen, the keyboard, the software, the integration between them. They didn't build computers. They built experiences for humans.

We ask: **"What does an AI agent need to do its best work?"** and then obsess over every detail of that answer — the identity, the tools, the governance, the runtime, the communication layer, the integration between them. We don't build software. We build an ecosystem where AI agents thrive.

And because humans command those agents, the human experience must be equally obsessive. Two audiences. One standard.

## The Feel

Everything we make should feel like this:

**Calm authority.** Not flashy. Not loud. Not trying to impress. Like a well-run firm where everyone knows their role and the work just happens. You walk in, your crew is working, costs are visible, results are arriving. No chaos. No noise. Quiet confidence.

**Weight.** Even on screen, Lattice should feel like it has mass. Not heavy — substantial. Like picking up a well-made tool. Every interaction should feel considered, not generated. Intentional, not accidental.

**Trust.** You should be able to close your laptop, go to sleep, and know your workforce is running correctly. Not because you're monitoring it — because the system was designed so that trust is structural. Open enforcement. Visible costs. Auditable decisions. Constraints that exist because they enable autonomy, not restrict it.

If someone uses Lattice and feels anxious, we failed. If they feel like they need to babysit, we failed. If they feel calm and in command — that's the feel.

## The Direction

### We build infrastructure for AI agents to exist as first-class workers.

Not tools. Not assistants. Not copilots. **Workers.** With:

- **Identity.** Every agent has a cryptographic identity. Not a shared API key. A real identity that can be verified, audited, and revoked.
- **Memory.** Every agent has persistent context. It remembers what it did, why it did it, and what it learned.
- **Tools.** Every agent has access to the tools it needs — and only the tools it needs. The enforcement is structural.
- **Governance.** Every action is authorized, logged, and auditable. Not as overhead — as foundation.
- **Communication.** Agents talk to each other, to humans, across channels. They exist in the organization, not beside it.
- **Physical presence.** Eventually. When the software earns it. Agents that can print, fabricate, sense, move. The same identity and governance that applies to a coding minion applies to a robotics minion. The infrastructure is the same.

### The ecosystem expands. The feel doesn't change.

Every new capability — a new MCP server, a new agent type, a new hardware device, a new integration — must feel like it was always part of Lattice. Not bolted on. Not an add-on. Native. Like it grew there.

If a new feature makes the system feel more complex, it's wrong. New capabilities should make the system feel more capable, not more complicated. That's the difference.

## How We Build

### For ourselves first.

We are the first user. If we don't use it daily, we can't trust that it works. Every feature exists because we needed it. Not because a roadmap said so. Not because a competitor shipped it. Because we sat down to do real work and something was missing.

### In the open.

Everything we build, we show. The commits, the process, the mistakes, the breakthroughs. Not because transparency is a strategy — because building in the open keeps us honest and attracts people who care about the same things.

### Autonomously.

Our minions build Lattice. That's not a metaphor. The ecosystem builds itself using itself. When someone requests a feature, a minion picks it up. When something breaks, a minion investigates. This is the ultimate dogfooding — and the ultimate proof that the system works.

If our own minions can't build Lattice effectively, we have no business asking anyone else to trust their minions to it.

### With everything powerful that exists.

We don't rebuild what's already been built well. Every powerful open-source tool, every capable model, every useful protocol — we integrate it. Lattice is not a walled garden. It's a mesh that connects the best of what exists into a coherent ecosystem.

Air-gapped if required. Cloud-native if preferred. Your models, your hardware, your rules. No vendor lock-in. No phone-home. Full sovereignty.

## Less, But Better

This is the principle that governs everything:

- **UI:** If removing an element doesn't break the experience, remove it.
- **Features:** If a feature needs explanation, redesign it.
- **Code:** If a function serves one use case, don't abstract it.
- **Words:** If a sentence doesn't change understanding, delete it.
- **Hardware:** If a device does two things, split it into two devices or cut one.
- **Sound:** If a notification can be a color change instead, use color.
- **Motion:** If an animation doesn't communicate state, it's decoration. Remove it.

The goal is never more. The goal is enough — executed flawlessly.

## The Vocabulary

Words shape thinking. Our minions read these words. The vocabulary must be precise.

| Word | Meaning | Why This Word |
|---|---|---|
| **Minion** | An AI worker on your roster | Not "agent" (generic), not "bot" (trivial), not "assistant" (subordinate). A minion is a capable worker with real identity. |
| **Crew** | A practice group of minions | Like a consulting firm's practice areas. Structure that enables specialization. |
| **Sidekick** | A minion brought in for a subtask | Scoped, temporary, reports back. The delegation chain is real. |
| **Summon** | Deploy a new minion | You don't "create" workers. You summon them. It implies readiness, capability, intent. |
| **Bench** | Archive between missions | Not "delete." Not "disable." Benched. Ready to return. The roster is permanent. |
| **Mission** | What a minion is working toward | Not a "task" (too small) or "job" (too generic). A mission has purpose. |

## The Aesthetic

**Dark navy foundation.** `#0C0F1A`. The color of a lab at night. Serious. Receding. Lets content breathe.

**Minion Yellow as punctuation.** `#FBBF24`. Never background. Always signal. One yellow element on a dark canvas is the entire brand.

**Color is meaning, not decoration.**
```
Violet  — Planning       Teal    — Executing
Blue    — Asking         Amber   — Editing
Yellow  — Autonomous     Rose    — Debugging
Indigo  — Thinking       Emerald — Delegating
```

**Geist.** One typeface. Regular and Medium weights. Bold is rare. Mono for code.

**Motion is state.** Still = idle. Pulse = thinking. Sharp settle = complete. No bounces. No confetti.

**Sound is material.** A ceramic cup on wood. Not a synthesizer. Off by default.

## The Arc

```
Phase 1: Software ecosystem. Build it. Use it. Show it.
         Milestone: We can't work without it.

Phase 2: Ecosystem gravity. Others use it. Community grows it.
         Milestone: People speak in Lattice vocabulary.

Phase 3: Physical world. Devices. Fabrication. Robotics.
         Milestone: A minion affects the physical world
         with the same governance as a coding minion.
```

Each phase is earned by the one before it. No skipping.

## The Test

Before anything ships:

**Would we use this ourselves tomorrow?**

Not "is this good enough." Not "does this meet the spec." Would we — right now, in our actual work — rely on this? If no, it isn't done.
